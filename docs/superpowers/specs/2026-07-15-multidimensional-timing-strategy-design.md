# 多维择时策略与报告设计

日期：2026-07-15
状态：用户已确认方向，等待书面规格复核

## 目标

把当前默认的“缩量择时”升级为通用的“基本面约束下的多维择时策略”。系统不读取个人持仓，不计算个性化仓位，只根据现有市场数据输出以下唯一主信号：

- `watch`：继续观察
- `low_buy`：适合低吸
- `accumulate`：适合抢筹
- `hold`：适合持有
- `reduce`：适合减仓
- `exit`：适合清仓

报告保留原有行情、均线、量能、筹码、新闻、评分和参考点位，但把“综合策略判断”放在最前面，并用多维证据说明理由。

## 非目标

- 不读取或推断用户实际持仓、成本和仓位比例。
- 不提供自动交易、下单或个性化投资建议。
- 不用单一成交量阈值、单一技术指标或固定回撤比例直接决定信号。
- 本轮不新增数据供应商，也不建设独立的确定性量化交易引擎。

## 判断层级

判断采用分层规则，不采用可以让重大风险被其他分数抵消的简单加权平均。

1. **数据质量门槛**：检查行情、K 线、财务和新闻的日期、完整性与可用性。数据缺失时降低置信度；关键信息缺失时不得输出高置信度的 `accumulate` 或 `exit`。
2. **长期逻辑过滤**：综合盈利与增长、现金流、负债、业绩预期、估值和行业地位。长期逻辑明显恶化时，超跌或缩量不能被解释为低吸机会。
3. **市场与行业环境**：结合大盘趋势、波动、行业强弱和个股相对强弱。高波动或行业显著转弱时，降低进攻型信号等级。
4. **价格结构与位置**：分析回撤、支撑压力、均线结构、形态、乖离和止跌/转强确认，确定当前处于低位、中位还是相对高位。
5. **量价、资金与筹码确认**：使用 5 日/20 日量能、上涨和下跌日量能差异、资金流和筹码作为确认，不单独决定信号。
6. **事件风险否决**：重大业绩恶化、财务或治理风险、监管处罚、退市风险等优先于普通技术信号。

运行时优先使用分析上下文中已经预取的数据，只在缺失时调用现有工具，避免重复请求和不必要的 GitHub Actions 时间消耗。相关工具包括 `get_daily_history`、`analyze_trend`、`get_volume_analysis`、`get_stock_info`、`get_market_indices`、`get_sector_rankings`、`get_capital_flow`、`get_chip_distribution` 和 `search_stock_news`；不支持的市场数据作为缺失项处理。

## 信号定义

### 继续观察 `watch`

- 数据不足、关键维度冲突或当前没有合理风险收益位置。
- 下跌缩量但价格仍在持续创新低、尚未止跌。
- 超跌但长期逻辑或风险信息无法确认。

### 适合低吸 `low_buy`

- 长期逻辑没有明显恶化，且不存在重大风险否决。
- 价格接近可验证支撑或处于合理回撤区域，跌幅和波动开始收敛。
- 量能、资金、形态或相对强弱中至少有一项支持抛压减弱。
- 尚未形成足够强的转强确认，因此强度低于 `accumulate`。

### 适合抢筹 `accumulate`

- 满足低吸的长期逻辑和风险条件。
- 出现明确止跌或转强确认，例如收复关键均线、形成更高低点、突破短期结构或相对强度改善。
- 量能温和恢复、资金改善、行业环境支持等至少一项提供独立确认。
- 价格不能已经明显远离支撑或处于追涨位置。

### 适合持有 `hold`

- 长期逻辑和主要价格结构保持正常。
- 当前既没有足够的低吸/抢筹条件，也没有明确的减仓/清仓条件。
- 允许存在短期震荡，但风险尚未升级。

### 适合减仓 `reduce`

- 价格接近压力、处于相对高位或乖离明显扩大。
- 同时出现至少一个独立转弱证据，例如上涨缩量、动能衰减、资金流出、行业转弱或利好兑现。
- 纯技术破位但长期逻辑尚未失效时，优先输出减仓而不是清仓。

### 适合清仓 `exit`

- 基本面或长期投资逻辑明确反转，或出现重大硬风险。
- 或长期结构严重破坏，并得到业绩、行业、资金或事件风险的独立确认。
- 单日下跌、单一均线破位或单一量能异常不得独立触发清仓。

## 冲突处理

- 超跌与缩量不能覆盖基本面恶化：输出 `watch`、`reduce` 或 `exit`。
- 下跌缩量但没有止跌结构：输出 `watch`，不能输出 `low_buy`。
- 上涨缩量但距离压力位较远且长期逻辑强：输出 `hold`，不能机械输出 `reduce`。
- 市场高波动、行业显著转弱或数据不完整：进攻信号至少降低置信度，必要时从 `accumulate` 降为 `low_buy` 或 `watch`。
- 多个维度结论冲突：选择风险更低的信号，并在理由中明确冲突，不强行给出方向性结论。

## 与现有动作和评分兼容

现有 `sentiment_score` 表示当前择时机会/风险，不代表公司长期质量。新信号映射到兼容字段：

| 新信号 | 分数区间 | `action` | `decision_type` |
|---|---:|---|---|
| `exit` | 0-19 | `sell` | `sell` |
| `reduce` | 20-39 | `reduce` | `sell` |
| `watch` | 40-49 | `watch` | `hold` |
| `hold` | 50-59 | `hold` | `hold` |
| `low_buy` | 60-79 | `buy` | `buy` |
| `accumulate` | 80-100 | `buy` | `buy` |

这组映射保持现有统计、历史记录和动作守卫逻辑兼容，同时由新的六态信号承担主要展示语义。

## 结构化输出契约

在 `dashboard` 下新增可选对象 `strategy_signal`：

```json
{
  "signal_code": "watch|low_buy|accumulate|hold|reduce|exit",
  "signal_label": "继续观察/适合低吸/适合抢筹/适合持有/适合减仓/适合清仓",
  "confidence": "高/中/低",
  "summary": "一句话综合结论",
  "reasons": [
    "[基本面] 具体结论与数据依据",
    "[价格位置] 具体结论与数据依据",
    "[量价资金] 具体结论与数据依据"
  ],
  "upgrade_trigger": "升级为更积极信号的可验证条件",
  "downgrade_trigger": "降级为观察、减仓或清仓的可验证条件"
}
```

约束：

- `reasons` 输出 3-5 条，至少覆盖三个独立维度。
- 每条理由必须引用实际数据、结构或已检索事件，禁止只写“趋势较好”等空泛结论。
- 数据缺失必须明确说明，不得补造数字。
- `signal_code`、评分、`action` 和 `decision_type` 必须遵守映射表。

## 报告展示

Markdown、飞书/微信和简版报告优先展示 `strategy_signal`。详细报告顺序为：

1. 综合策略信号、置信度和一句话结论。
2. 按维度标记的判断理由。
3. 升级条件和降级/失效条件。
4. 原有新闻情报、行情快照、均线结构、价格位置、量能、筹码和阶段决策。
5. 原有参考点位、检查清单和信号贡献。

原有 `trend_prediction` 和 `sentiment_score` 继续作为辅助信息展示。信号贡献区的展示名称从“最强看多/看空信号”调整为“主要支持因素/主要风险因素”，底层兼容字段不变。没有 `strategy_signal` 的旧报告继续使用原模板行为。

## 验证标准

- 默认激活技能仍只有 `volume_contraction_timing`。
- 两套提示词都要求六态信号、多维理由和风险否决。
- Schema 能解析完整和缺省的 `strategy_signal`。
- 三种报告在有新信号时优先展示它，在无新信号时保持原有输出。
- 报告仍包含原有行情、均线、量能、新闻、评分和参考点位。
- 信号、评分、动作和三态统计没有冲突。
- 相关单元测试、报告模板测试和默认技能测试全部通过。

## 后续校准

本轮不把未经回测的具体阈值固化为绝对规则。信号运行积累样本后，可分别统计六类信号未来 5、10、20、60 个交易日的收益、回撤和命中率，再按市场、波动环境和股票类型校准阈值。
